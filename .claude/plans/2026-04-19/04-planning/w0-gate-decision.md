## 2. W0 결과 인지 사항

### 2.1 T-W0-01 Anthropic Cookbook — 🟡 부분 훼손

- **Cookbook 본문 자체**: 6축 메타 레이어 명시 0건. "harness"·"meta-framework" 용어 미등장.
- **Anthropic 공식 주변**: "Harness"를 공식 용어로 사용. "Three-Agent Harness" (계획/실행/검증 3축 분리)가 `patterns/agents/evaluator_optimizer.ipynb`에서 이미 4축(계획/실행/검증/개선) 루프 표면화 → **우리 6축 중 3~4축은 Anthropic 공식 선행 레퍼런스에 존재**.
- **커뮤니티 `ai-boost/awesome-harness-engineering`**: 12 primitive 카테고리로 이미 세분화된 체계 공개. 6축 → 12축으로 오히려 더 깊게 분류됨.
- **유일 공백**: **개선(Compounding) 축** — Cookbook·Anthropic 공식·12축 분류 어디에도 "세션 간 자동 학습 축적" 레이어 없음.

### 2.2 T-W0-02 4-framework 비교 — 🟢 강화 + 🟡 축 깊이 리스크

| 프레임워크 | 메타 레이어 명시? | 개선 축 |
|-----------|:--------------:|:------:|
| DSPy v3.1.3 | No (Optimizer 중심) | △ (compile-time만, 세션 간 아님) |
| Inspect-AI v0.3.199 | No (Evaluation 중심) | △ (수동 Eval Sets) |
| LangGraph v1.1.6 | No (Orchestration 중심) | ❌ |
| AutoGen v0.4 / MS Agent FW 1.0 | No (Multi-Agent 중심) | ❌ |

- **강화**: 4/4 프레임워크 모두 6축 메타 레이어를 표면화하지 않음 → "6축을 묶는 메타 레이어 부재" 프리미스 **성립**.
- **축 깊이 리스크**: 개별 축은 4 프레임워크가 우리보다 깊게 구현. 우리는 개별 축을 새로 만들지 말고 **메타 레이어(통합)만 새로 만들어야 함**.
- **개선 축**: 4/4 전부 공백 → **유일 고유 차별화 지점**. (참고로만 보존, TL;DR 변경 없음.)

### 2.3 종합 인지

- **"6축 메타 레이어 부재" 프리미스**: Anthropic 공식·커뮤니티가 3축·12축을 선행 표면화했으나, **6축을 하나로 묶어 구조적으로 강제하는 공용 플러그인은 여전히 부재** → 프리미스 전면 훼손 아님.
- **개선(Compounding) 축의 공백**: 현 시점에서도 모든 조사 범위에서 유효 → 2차 릴리스 또는 dogfooding 실증 단계에서 강조 자산으로 활용 가능 (참고 메모로 보존).

## 3. 판정 근거

1. **프리미스 유지**: 6축 × 공용 플러그인 × 승격 게이트라는 **조합**은 W0 조사 범위에서 선행 사례 0건. 개별 축 깊이 리스크는 포팅 매트릭스(`porting-matrix.md`)의 "최선 패턴 차용" 전략으로 이미 완화.
2. **재정의 비용 > 실행 모멘텀 손실**: TL;DR 문장을 "개선 축 중심"으로 좁히면 `final-spec.md` §1 + §2 + §6·§7 포팅 자산 재정합 필요. user-decisions-5 §5 포지셔닝 2문장은 **재구성 없이 그대로 MVP 릴리스 기준 유지**하는 편이 W1 착수를 지연시키지 않는다.
3. **플러그인 레이어는 여전히 공백**: Anthropic Cookbook·4 프레임워크·12축 커뮤니티 어디에도 `.claude-plugin/` 형태의 공용 배포 자산 부재. → 배포 층의 차별화는 포지셔닝 변경 없이도 성립.
4. **개선 축 공백은 보너스 자산**: TL;DR 수정 없이도, W6 `/compound` 구현 단계에서 "개선 축 메카닉이 타 레퍼런스에 부재"임을 README·marketing에서 추가로 강조 가능. 설계 차원 변경 불필요.
